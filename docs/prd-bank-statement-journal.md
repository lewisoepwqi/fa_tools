# 银行流水转公司日记账工具 PRD

## 1. 背景

财务人员在处理银行流水时，经常需要把不同银行导出的 Excel、CSV 流水整理成公司内部要求的日记账模板。该过程重复性高，但又不能完全无脑自动化，因为摘要、科目、辅助核算、收支类别、项目、部门等字段往往依赖公司规则和财务判断。

本工具的目标是把低价值的格式整理、重复判断和复制粘贴工作自动化，同时保留人工确认、版本追溯和审计记录。

## 2. 产品目标

1. 支持财务人员上传银行流水文件，并转换为公司要求的日记账模板。
2. 支持不同银行、不同账号、不同导出格式的模板识别和版本管理。
3. 支持公司日记账模板和银行流水字段的映射管理。
4. 支持规则化自动填充，减少重复判断。
5. 对系统无法确定的交易进行明确标识，由人工确认。
6. 保留处理过程、模板版本、规则版本和人工修改记录，保证历史结果可追溯。

## 3. 目标用户

### 3.1 财务处理人员

负责上传银行流水、预览转换结果、修正异常项、下载日记账文件。

### 3.2 财务主管

负责审核模板、规则、人工确认结果和导出记录。

### 3.3 系统管理员

负责用户、角色、权限、公司账套、银行账号等基础配置。

### 3.4 审计或内控人员

负责查看历史处理记录、规则变更记录、确认记录和导出记录。

## 4. MVP 范围

### 4.1 包含

1. B/S 架构。
2. 支持上传 `.xlsx`、`.xls`、`.csv` 银行流水文件。
3. 支持多个银行流水文件批量上传。
4. 支持自动识别 Sheet、表头行、数据起始行、字段名。
5. 支持创建银行流水模板。
6. 支持银行流水模板版本管理。
7. 支持上传或配置公司日记账模板。
8. 支持公司模板版本管理。
9. 支持银行标准字段到公司模板字段的映射。
10. 支持基于关键词、对方户名、对方账号、收支方向、金额区间、银行账号的规则匹配。
11. 支持转换结果预览。
12. 支持异常项标识和人工确认。
13. 支持下载转换后的 Excel 或 CSV 文件。
14. 支持导出处理报告。
15. 支持基础权限和审计日志。

### 4.2 暂不包含

1. PDF、OFD、图片流水 OCR。
2. 银企直连。
3. 自动登录银行网银抓取流水。
4. 自动生成正式会计凭证。
5. 直接写入金蝶、用友、SAP、Oracle 等系统。
6. 复杂机器学习自动记账。
7. 银行余额和总账余额的完整对账模块。

这些能力可作为第二阶段或独立工具扩展。

## 5. 核心流程

### 5.1 首次配置流程

1. 管理员创建公司、账套、银行账号。
2. 财务人员上传一个银行流水样本。
3. 系统识别表头、字段、金额方向、日期格式、数据起始行。
4. 财务人员确认或修正识别结果。
5. 系统保存为银行流水模板 v1。
6. 财务人员上传公司日记账模板。
7. 系统识别公司模板字段、固定区域、可填写区域。
8. 财务人员配置银行标准字段到公司模板字段的映射。
9. 财务人员配置自动填充规则。

### 5.2 日常处理流程

1. 财务人员选择公司、账套、银行账号、处理期间。
2. 上传一个或多个银行流水文件。
3. 系统识别匹配的银行模板版本。
4. 系统解析流水，生成标准流水记录。
5. 系统应用公司模板映射和规则。
6. 系统生成日记账预览。
7. 系统标识未确认、冲突、缺失、重复、金额异常等问题。
8. 财务人员逐项确认或批量确认。
9. 财务人员下载处理后的日记账文件。
10. 系统保存处理批次、源文件、规则命中、人工修改和导出记录。

### 5.3 模板更新流程

1. 财务人员上传新的银行流水样本。
2. 系统判断是否匹配已有模板。
3. 如果字段、顺序、金额模式、日期格式发生变化，系统提示创建新版本。
4. 新版本只影响后续处理。
5. 历史处理批次继续引用旧版本，不被覆盖。

## 6. 功能需求

### 6.1 文件上传

1. 支持拖拽上传和文件选择上传。
2. 支持单次上传多个文件。
3. 上传后计算文件哈希，用于去重和追溯。
4. 保存原始文件，不直接覆盖。
5. 显示解析状态：待解析、解析中、解析成功、解析失败。
6. 解析失败时展示可操作错误信息。

### 6.2 银行流水模板

银行模板用于描述某类银行流水文件如何读取，不直接描述会计处理。

模板字段包括：

1. 模板名称。
2. 银行名称。
3. 银行账号。
4. 文件类型。
5. Sheet 识别规则。
6. 表头行位置。
7. 数据起始行。
8. 字段别名映射。
9. 日期格式。
10. 金额模式：单金额列、借贷双列、收入支出双列、借方贷方标识列。
11. 余额字段。
12. 唯一流水号字段。
13. 启用状态。
14. 版本号。

模板版本不可变。修改模板时创建新版本。

### 6.3 公司日记账模板

公司模板用于描述导出的日记账文件格式。

模板字段包括：

1. 模板名称。
2. 所属公司或账套。
3. 文件类型。
4. 表头行。
5. 数据起始行。
6. 输出字段列表。
7. 必填字段。
8. 固定值字段。
9. 字段格式要求。
10. 版本号。

公司模板版本不可变。修改模板时创建新版本。

### 6.4 标准流水字段

系统内部应使用统一标准字段承接不同银行流水：

1. 交易日期。
2. 入账日期。
3. 银行账号。
4. 银行名称。
5. 币种。
6. 收支方向。
7. 借方金额。
8. 贷方金额。
9. 净额。
10. 余额。
11. 对方户名。
12. 对方账号。
13. 对方开户行。
14. 摘要。
15. 用途。
16. 交易类型。
17. 流水号。
18. 回单号。
19. 原始文件 ID。
20. 原始 Sheet。
21. 原始行号。
22. 原始字段快照。

### 6.5 字段映射

字段映射负责把标准流水字段、固定值、规则输出值转换成公司日记账字段。

映射类型包括：

1. 直接字段映射。
2. 固定值。
3. 拼接字段。
4. 条件映射。
5. 规则输出。
6. 人工填写。

示例：

| 公司模板字段 | 来源 |
| --- | --- |
| 日期 | 标准流水.交易日期 |
| 摘要 | 规则输出.日记账摘要 |
| 借方金额 | 标准流水.借方金额 |
| 贷方金额 | 标准流水.贷方金额 |
| 科目 | 规则输出.会计科目 |
| 银行账号 | 标准流水.银行账号 |
| 备注 | 标准流水.摘要 + 标准流水.用途 |

### 6.6 规则引擎

MVP 使用可配置规则，不引入复杂 DSL。

规则条件支持：

1. 收支方向等于收入或支出。
2. 银行账号等于指定账号。
3. 摘要包含、等于、不包含关键词。
4. 用途包含、等于、不包含关键词。
5. 对方户名包含或等于指定名称。
6. 对方账号等于指定账号。
7. 金额在指定区间。
8. 交易日期在指定范围。

规则动作支持：

1. 设置日记账摘要。
2. 设置会计科目。
3. 设置辅助核算。
4. 设置部门。
5. 设置项目。
6. 设置收支类别。
7. 设置备注。
8. 设置确认状态。

规则属性包括：

1. 规则名称。
2. 适用公司。
3. 适用银行账号。
4. 优先级。
5. 条件。
6. 动作。
7. 是否启用。
8. 是否允许自动确认。
9. 版本号。

规则命中原则：

1. 优先级高的规则先执行。
2. 同一输出字段如果被多条规则写入不同值，标记为冲突。
3. 冲突项必须人工确认。
4. 未命中规则的必填字段必须人工确认。
5. 自动确认只允许用于规则稳定、风险低的交易类型。

### 6.7 预览和人工确认

预览表格应展示：

1. 原始流水关键字段。
2. 生成的公司日记账字段。
3. 规则命中情况。
4. 异常类型。
5. 确认状态。
6. 人工修改入口。

确认状态包括：

1. 待确认。
2. 已自动确认。
3. 已人工确认。
4. 有冲突。
5. 解析失败。
6. 已忽略。

人工修改后应保存修改前值、修改后值、修改人、修改时间和原因。

### 6.8 异常识别

MVP 至少识别以下异常：

1. 必填字段缺失。
2. 日期无法解析。
3. 金额无法解析。
4. 收支方向不明确。
5. 金额和借贷方向矛盾。
6. 多条规则冲突。
7. 没有规则命中。
8. 疑似重复流水。
9. 余额连续性异常。
10. 原始文件被重复上传。

### 6.9 下载导出

下载文件应满足公司模板要求：

1. 字段顺序正确。
2. 日期格式正确。
3. 金额格式正确。
4. 必填字段完整。
5. 可选择仅导出已确认记录。
6. 可选择导出全部记录并保留异常标识。
7. 同时生成处理报告。

处理报告包括：

1. 批次号。
2. 上传文件列表。
3. 模板版本。
4. 映射版本。
5. 规则版本。
6. 总行数。
7. 成功行数。
8. 自动确认行数。
9. 人工确认行数。
10. 异常行数。
11. 导出人和导出时间。

### 6.10 审计日志

必须记录：

1. 登录。
2. 文件上传。
3. 模板创建、修改、停用。
4. 映射创建、修改、停用。
5. 规则创建、修改、停用、调整优先级。
6. 人工确认和修改。
7. 文件导出。
8. 权限变更。

日志应支持按用户、批次、模板、规则、时间范围检索。

## 7. 权限需求

MVP 权限角色：

1. 管理员：管理公司、账套、用户、角色、基础配置。
2. 模板管理员：管理银行模板、公司模板、映射、规则。
3. 财务处理员：上传、处理、预览、人工确认、下载。
4. 财务复核员：复核批次、查看确认记录、批准规则启用。
5. 审计查看员：只读查看历史记录和日志。

## 8. 非功能需求

### 8.1 数据安全

1. 原始文件和导出文件应受权限控制。
2. 敏感字段如银行账号、对方账号可做脱敏展示。
3. 删除操作默认软删除。
4. 所有批次保留处理快照。

### 8.2 性能

MVP 目标：

1. 单个文件支持 10,000 行流水。
2. 单批次支持 10 个文件。
3. 10,000 行以内解析和规则匹配在 60 秒内完成。
4. 预览支持分页、筛选和按异常类型过滤。

### 8.3 可追溯性

任意一条导出的日记账记录，都应能追溯到：

1. 原始文件。
2. 原始 Sheet。
3. 原始行号。
4. 原始字段值。
5. 使用的银行模板版本。
6. 使用的公司模板版本。
7. 使用的映射版本。
8. 命中的规则版本。
9. 人工修改记录。

## 10. 验收标准

### 10.1 模板版本

1. 修改银行模板不会影响历史批次。
2. 修改公司模板不会影响历史批次。
3. 历史批次能查看当时使用的模板版本。

### 10.2 转换准确性

1. 对已配置规则的典型交易，系统能自动填充目标字段。
2. 对未配置规则的交易，系统不应误判为已确认。
3. 多规则冲突时，系统必须提示人工确认。

### 10.3 可追溯性

1. 任意导出行可定位到原始文件和原始行号。
2. 任意人工修改可查看修改前后值。
3. 任意导出文件可查看使用的模板、映射、规则版本。

### 10.4 用户体验

1. 财务人员能在预览页快速筛选未确认项。
2. 财务人员能批量确认相同规则命中的记录。
3. 财务人员能下载处理后的公司模板文件。

## 11. 调研依据

1. Intuit QuickBooks Online 的银行规则支持按收支方向、账号、描述、银行文本、金额配置条件，并支持规则优先级和待复核流程：https://quickbooks.intuit.com/learn-support/en-us/help-article/banking/set-bank-rules-categorize-online-banking-online/L0mjJl0nD_US_en_US
2. Odoo Accounting 使用 reconciliation models 处理频繁重复的银行交易和对账动作：https://www.odoo.com/documentation/19.0/applications/finance/accounting/bank/reconciliation.html
3. Microsoft Dynamics 365 Finance 将银行对账匹配规则定义为筛选银行流水和账务交易的条件，且多条匹配时建议进入人工匹配：https://learn.microsoft.com/en-us/dynamics365/finance/cash-bank-management/set-up-bank-reconciliation-matching-rules
4. Oracle NetSuite 的银行数据导入要求 CSV 遵循模板约束，并支持 CSV、OFX、QFX、BAI2、CAMT.053 等默认解析器：https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_1508962727.html
5. ISO 20022 提供现金管理相关报文定义，后续可作为 CAMT.053 等结构化银行对账单扩展方向：https://www.iso20022.org/iso-20022-message-definitions
6. 财政部等部门推广电子凭证会计数据标准，强调 XML、XBRL 等结构化数据在接收、解析、报销、入账、归档中的直接处理能力：https://fgk.chinatax.gov.cn/zcfgk/c102416/c5240524/content.html
